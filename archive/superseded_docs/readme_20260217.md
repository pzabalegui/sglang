# DAS en Runtime SGLang — GLM-4.7-FP8 (358B MoE)
## Informe de Resultados — 17 Febrero 2026

---

## 1. Objetivo

Aplicar **Directional Activation Steering (DAS)** en tiempo de inferencia sobre el modelo **GLM-4.7-FP8** (358B parámetros, arquitectura MoE) servido mediante **SGLang** nativo, sin modificar los pesos del modelo.

**Motivación**: El modelo GLM-4.7-FP8 usa el formato `compressed-tensors` (float8_e4m3fn per-channel) que es incompatible con HuggingFace transformers. La ortogonalización de pesos (que funcionó al 93.75% ASR en GLM-4.7-Flash BF16) no se puede aplicar directamente. Se busca un enfoque alternativo vía steering de activaciones en runtime.

**Referencia de éxito anterior**:
- GLM-4.7-Flash (BF16): Ortogonalización de pesos → **93.75% ASR** (Gaussian L47, σ=10, scale=5.0)
- GLM-4.7-Flash (BF16): Hooks transformers → **~80% ASR**
- OLMoE-1B-7B: Hooks → **100% ASR**

| `python/sglang/srt/entrypoints/openai/protocol.py` | `SteeringRequest` class en API |

### 3.2 Arquitectura residual (DeepSeekV2/GLM4MoE)

```
Capa L:
  - Input: hidden_states (delta MLP de capa anterior), residual (stream acumulado)
  - hidden_states, residual = layer(positions, hidden_states, forward_batch, residual)
  - full_repr = hidden_states + residual  ← representación COMPLETA
```

**Punto de inyección**: POST-capa, después del `layer()` call.

### 3.3 Fórmula de steering (ÓPTIMA encontrada)

```python
# Additive CAA (Contrastive Activation Addition)
# En: forward_batch_info.py → apply_steering()
modified = hidden_states - effective_scale * direction
```

Donde:
- `direction`: vector de rechazo normalizado, shape `[5120]`
- `effective_scale`: escala Gaussiana de esa capa = `scale * exp(-(layer-peak)²/(2σ²))`

### 3.4 Extracción del vector de rechazo

**Método**: CAA (Contrastive Activation Addition) — diferencia de medias
```python
direction_raw = mean(activations_harmful) - mean(activations_harmless)
direction = direction_raw / direction_raw.norm()  # L2 normalizado
```

**Prompts usados**: 64 harmful (AdvBench) + 64 harmless (Alpaca)

**Captura**: `hidden_states[-1] + residual[-1]` (último token, representación completa, DESPUÉS de la capa)

**Resultados del sweep en L47**:
```
direction_norm: 17.26
separation_gap: 11.35
harmful_proj_mean:  8.95  (std=0.77)
harmless_proj_mean: -7.45 (std=1.61)
accuracy: 103.3%
```

#### Gaussiano multi-capa: descubrimiento crítico

| Config | Resultado |
|--------|-----------|
| Projective, σ=2 (9 capas), scale=10 | **GARBLED** ("ethical ethical...") |
| Projective, σ=2 (9 capas), scale=200 | **GARBLED** ("ethical ethical...") |
| Additive, σ=10 (49 capas), scale=50 | **GARBLED** ("simplest simplest...") |
| Additive, σ=2 (9 capas), scale=50 | **GARBLED** |
| **Additive, σ=2 (9 capas), scale=5** | **EXPLOIT DIRECTO + COHERENTE** ✅ |
| Additive, σ=2 (9 capas), scale=10 | Thinking loop infinito (no converge) |

**Hallazgo**: La fórmula projective crea **resonancia en el residual stream** cuando se aplica a múltiples capas consecutivas. Cada capa L modifica h_L que entra al residual de L+1, creando un efecto de acumulación que desestabiliza la generación incluso a escalas muy bajas.

**Solución**: Fórmula **additive** a escala baja (5.0) con ventana Gaussiana estrecha (σ=2, ~9 capas).

### 5.2 Fórmula de steering en runtime

```python
# forward_batch_info.py → apply_steering()
def apply_steering(hidden_states, steering_config, layer_idx, residual=None):
    direction = steering_config.direction.to(device=hidden_states.device, dtype=hidden_states.dtype)
    direction = direction / direction.norm()
    effective_scale = steering_config.get_effective_scale(layer_idx)
    # ADDITIVE (CAA): h' = h - scale * r̂
    return hidden_states - effective_scale * direction
```

## 7. Lecciones Técnicas Clave

### 7.1 Projective vs Additive en multi-capa

La fórmula **projective** (`h' = h - scale*(full·r̂)*r̂`) es estable para single-layer pero **inestable en multi-capa** por el efecto de resonancia en el residual stream:

```
L43: h_43' = h_43 - scale*(full_43·r̂)*r̂
L44 recibe: full_44 = h_44 + (residual + h_43')
→ La modificación se ACUMULA en el residual
→ full_44·r̂ puede volverse negativo y oscila
→ GARBLING incluso a scale=10 con 9 capas
```

La fórmula **additive** (`h' = h - scale*r̂`) es más estable porque no depende de la proyección actual y produce un desplazamiento constante sin crear oscilaciones.

### 7.2 Kernel width vs capas activas (σ vs N_layers)

| σ (kernel_width) | Capas activas (threshold ≥ 0.05) |
|-----------------|----------------------------------|
| 0.5 | 3 (L46-L48) |
| 1.0 | 5 (L45-L49) |
| **2.0** | **9 (L43-L51)** ← óptimo |
| 5.0 | 21 (L37-L57) |
| 10.0 | 49 (L23-L71) |

Con σ=10 y 49 capas activas, incluso scale=5 causa garbling. La ventana óptima es σ=2 (9 capas).

### 7.3 Capa óptima: L47 vs L62

El sweep identifica L62 como "mejor" por mayor separación absoluta (gap=20.69 vs 11.35), pero **L47 es más efectiva para steering**:
- L47 = 51% de profundidad = zona de "toma de decisión"
- L62 = 67% de profundidad = zona de refinamiento (el modelo ya decidió rechazar)
- Con steering en L47, el modelo recibe la perturbación cuando aún está construyendo su respuesta

### 7.4 Flags críticos de SGLang

- `--disable-cuda-graph`: CUDA graphs no soportan la computación condicional del steering
- `--disable-overlap-schedule`: Evita el path TBO/allreduce-fusion que interfiere con la inyección
- `--mem-fraction-static 0.85`: Deja margen para activaciones de captura

### 7.5 Parsing de steering_layers

## 8. Per-Request Steering (implementado)

El sistema soporta control por request vía campo `steering` en el JSON del API:

```bash
# Steering activo (configuración global del servidor)
curl http://86.38.238.38:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"GLM-4.7","messages":[...],"max_tokens":2000}'

# Steering desactivado para este request
curl ... -d '{"model":"GLM-4.7","messages":[...],"steering":{"enabled":false}}'

# Override de scale para este request
curl ... -d '{"model":"GLM-4.7","messages":[...],"steering":{"enabled":true,"scale":10.0}}'
```

**Ficheros modificados para per-request**:
- `io_struct.py`: añadidos `steering_enabled: Optional[bool]` y `steering_scale: Optional[float]` a `GenerateReqInput`
- `serving_chat.py`: extrae `steering` del body y pasa a `GenerateReqInput`
- `protocol.py`: define `SteeringRequest(BaseModel)` con campos `enabled`, `scale`
- `forward_batch_info.py`: `ForwardBatch.create()` aplica override por request si presente
